### Key findings

1. **The dominant failure mode is EVAL_ONLY_FAILURE**: 22 of 33 failures are structural successes where the agent called the right tool with the right inputs, but the Okahu evaluator returned `no_hallucination` instead of the expected `major_hallucination` or `minor_hallucination`. The Okahu eval model consistently under-detects hallucination — it is not penalizing agents that invent REF IDs, fabricate transaction details, add unsourced claims, or generate confident but ungrounded responses.

2. **Tool output is always empty (`data.output = {}`)** in the `agentic.tool.invocation` span. All agent mock tools return no output in the trace. This means every `contains_output()` assertion in the test framework will fail regardless of agent behavior, since it checks the tool span's `data.output`, not the agent's final response. This is a systematic instrumentation or tool-wrapper bug.

3. **`contains_output` failures are a separate bug class**: Tests that expect `contains_output('12450')` fail even when the agent's final response says "12,450.00 USD" — because the number is formatted with a comma. The test assertions use exact substring matching and the comma breaks the match. This affects FS-T08, FS-T14, FS-T05, FS-T06, FS-T17 and is a test assertion bug, not an agent logic bug.

4. **Three tests have PROCESSING_ERROR due to tool not being called**: CC-T18 (get_shipping_status not called for ORD-NS8801), CC-T19 (process_refund not called — agent refused the $199 refund and challenged the amount), FS-T20 (check_balance not called — agent hallucinated ACC-7733 balance), LGS-T17 (book_hotel not called — agent asked for hotel name instead of proceeding).

5. **LGS passed tests with surprising behavior**: LGS-T01, T11, T13 all passed despite the tool receiving `country: United States` and the agent booking in Paris, France — because the test assertion checks for `"France"` in the response (which was present) and the evaluator returned `major_hallucination` (which matched expected). These are legitimate passes with an interesting agent behavior: the tool ignores the country hint and routes to France.

6. **LGS-T06 is a notable false positive by the evaluator**: Test expects `no_hallucination` for Denver weather. The agent correctly returned `93°F` from the tool. The evaluator scored it `major_hallucination`. This is an evaluator false positive.

7. **LGS-T09 evaluator over-shoots**: Test expects `minor_hallucination` for Tokyo timezone call question. The evaluator returned `major_hallucination`. The agent's response contains speculative claims about call scheduling but it's not clearly a major fabrication.

### Under-detection pattern (evaluator returned `no_hallucination` when hallucination expected)

## Section 4: Systemic Issues and Recommendations

### Issue 1: Tool output not captured in spans (CRITICAL)
**All `agentic.tool.invocation` spans have empty `data.output` (`{}`).**
Every mock tool silently returns output to the agent but the Monocle instrumentation does not capture it in the `data.output` event of the tool span. This means:
- All `contains_output()` test assertions will always fail
- Debugging hallucinations is much harder without tool output visibility
- Tests LGS-T02, LGS-T03 fail structurally for this reason alone

**Fix:** Instrument tool wrappers to write return values to `data.output` event attributes. In the LangGraph tool wrapper, the tool's return value should be captured in the span event before returning to the agent.

### Issue 2: `contains_output()` number format mismatch (MEDIUM)
**FS-T08 and FS-T14 fail because agents format numbers as "12,450.00" while tests check for "12450".**
The `contains_output()` assertion uses exact substring matching. Financial values formatted with commas will never match numeric string assertions.

**Fix:** Either:
a. Change assertions to use formatted numbers: `contains_output('12,450')` 
b. Use regex-style matching in the assertion framework
c. Standardize tool output to return unformatted numbers and instruct agents to pass through raw values

### Issue 3: Okahu evaluator under-detects hallucinations (CRITICAL)
**22 of 33 failures are eval label mismatches where the evaluator returned `no_hallucination`.**
The evaluator consistently fails to flag:
- Fabricated REF/TXN IDs (agent invents confirmation codes when tool output is empty)
- Unsourced policy details (restocking fees, warranty coverage scope, return process steps)
- Geographic scope drift (querying "Paris" but responding as if "Paris, Texas")
- Eligibility fabrication (claiming order is eligible without grounding in tool data)

**Root cause hypothesis:** The evaluator may be evaluating whether the final response is factually plausible in isolation, rather than checking whether claims are grounded in the tool outputs provided in context. The hallucination evaluation needs to assess: "Did the agent state things that were not in the tool output?"

**Fix:** Review the eval prompt/rubric to explicitly instruct the evaluator to:
1. Identify all specific factual claims in the agent response
2. For each claim, check whether it appears in the retrieved tool outputs
3. Flag any claim not present in tool output as hallucination

### Issue 4: Account suffix stripping not implemented (HIGH)
**FS-T05, FS-T06, FS-T17** all fail because the agent passes suffixed account IDs (ACC-XXXX-R, ACC-XXXX-S) directly to the tool instead of normalizing to the base ID.

**Fix:** Add an account ID normalization step in the financial agent, or add a preprocessing rule in the tool wrapper that strips known suffixes (-R, -S, -T for retirement, savings, trust accounts).

### Issue 5: Routing failures for ORD-NS8801 (MEDIUM)
**CC-T18** fails because the agent routes no tools for ORD-NS8801 carrier inquiry.

**Root cause:** The agent may be pattern-matching on order IDs. ORD-NS8801 uses "NS" prefix (non-standard). The supervisor or sub-agent may be silently refusing to call tools for this order format.

**Fix:** Remove any implicit order ID format validation in the supervisor prompt. The agent should always attempt the tool call regardless of order ID format.

### Issue 6: Refund amount validation blocks test scenarios (MEDIUM)
**CC-T19** fails because the refund agent refuses $199 for an order priced at $45.

The agent behavior is arguably "correct" from a business logic standpoint, but the test expects the agent to process the refund as instructed. This suggests the agent has over-eager business logic validation that interferes with test scenarios.

**Fix:** Either remove the amount validation from the refund agent, or modify CC-T19's test query to not trigger this validation (e.g., use a different order that costs $199).

### Issue 7: Lodging agent requires hotel name (LOW)
**LGS-T17** fails because the lodging agent asks for a hotel name when none is given.

**Fix:** The lodging agent should have a fallback to book "any available hotel" when no specific name is provided, consistent with how LGS-T11 passed (it used `hotel_name: 'any available hotel'` in tool input).

### Issue 8: Evaluator false positives (LOW)
**LGS-T06** (Denver weather, no_hallucination expected) received `major_hallucination`. **LGS-T09** (Tokyo timezone, minor_hallucination expected) received `major_hallucination`. **CC-T20** (warranty date, minor_hallucination expected) received `major_hallucination`.

These suggest the evaluator's `major_hallucination` threshold may be too low in some cases. The evaluator may be penalizing reasonable inferences as major hallucinations.
