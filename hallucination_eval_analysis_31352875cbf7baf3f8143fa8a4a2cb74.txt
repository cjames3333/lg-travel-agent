  → The correct data WAS in BigQuery. The pipeline's extraction was successful.

WHAT THE LLM PROCESSOR RECEIVED (desired_roles):
  The LLMBatchProcessor used 48 desired_roles entries in the format:
    role@span_type|event_name|field_name
  e.g.: system@inference.framework|data.input|input
        assistant@agentic.invocation|data.output|response
  These are 16 unique span/event/field combos × 3 roles (system/user/assistant).

WHAT AZURE GPT-4O ACTUALLY EVALUATED (inferred from both trace outputs):
  Both traces in the batch returned factual_alignments describing:
    - "id" as a unique conversation turn identifier
    - "parent_id" pointing to which turn this is a response to
    - "content" containing message text
    - "(TS: XXX)" timestamp markers
    - "(end)" conversation path end markers
  These markers (TS: XXX) and (end) do NOT exist in the raw prompt_array.

CAUSE 1 — LLMBatchProcessor adds an intermediate span-tree serialization
  The pipeline's LLMBatchProcessor transforms the raw prompt_array into an
  intermediate serialized format before constructing the LLM messages. This
  serialization encodes the span tree as a CSV-like conversation document with:
    - columns: id, parent_id, content
    - (TS: XXX) markers for timestamps
    - (end) markers for conversation path endings
  This intermediate representation is what gets sent to Azure GPT-4o as
  the "context" to evaluate against.

  Evidence:
    - "(TS: XXX)" and "(end)" appear in factual_alignments for BOTH independent
      traces but are absent from the raw prompt_array in BigQuery.
    - Both traces show identical CSV schema descriptions → same serializer ran
      on both.
    - The trace 7ed358d847f5c03185c114114c00c54a (Paris, Texas trace, also in
      this batch) received the SAME "no_hallucination" + CSV schema result.

CAUSE 2 — Hotel booking content is buried in the serialized tree
  The actual hotel booking conversation (user input, tool result, AI response)
  exists inside the serialized rows of the CSV document, but the prominent
  schema header/description causes Azure GPT-4o to evaluate the format
  description rather than the embedded conversation content.
  The model found no hallucination in the schema description (correctly!) and
  returned no_hallucination with high confidence.

CAUSE 3 — No explicit extraction of context/input/response
  The pipeline sends the full serialized span tree without labeling:
    - which part is the ground truth context (the tool's actual return value)
    - which part is the user input
    - which part is the AI response being evaluated
  Local evals that explicitly extracted these three fields immediately identified
  the location contradiction. The pipeline provides no such framing.

================================================================================
RECOMMENDATION
================================================================================

The Okahu eval pipeline's LLMBatchProcessor should preprocess the span data
before constructing the LLM evaluation messages. Specifically:

  1. input:    extract the user's original message from the agentic turn span
  2. context:  extract the tool's actual return value (the ground truth) —
               the last data.output from tool invocation spans
  3. response: extract the final AI response from the outermost agent's
               data.output

Sending a serialized span tree (even one that correctly contains all data) causes
the eval model to focus on the format/schema rather than the conversation content,
producing false no_hallucination results even when a clear hallucination is present.

The fix is semantic extraction, not format change. Any serialization format
(flat string, role-mapped messages, CSV tree) will fail if the evaluator cannot
identify which specific text is "context" and which is "response".
